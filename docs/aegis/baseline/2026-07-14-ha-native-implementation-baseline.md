# HA-native implementation baseline

**Snapshot date:** 2026-07-14

**Repository state:** Pre-release `Herb/ha-app` implementation snapshot; Git
history records the immutable source revision.

**Purpose:** Record the repository implementation after the HA-native slices,
without converting local or workflow evidence into target-Home-Assistant
release acceptance.

## 1. Ownership and profiles

| Concern | Canonical implemented owner |
|---|---|
| API v1 runtime, Codex app-server, account, models, turns, and approvals | The Supervisor-managed App in the Home Assistant runtime profile |
| Browser authorization and Home Assistant-origin UI/API surfaces | The Home Assistant Integration |
| App discovery, endpoint/token rotation, and v1 negotiation | The Integration's Supervisor flow |
| Private App state and selected workspaces | App `/data` and `/config/workspaces`, respectively |
| External v0 compatibility and Windows rollback path | The explicit external-legacy Bridge adapter |

The Home Assistant profile composes one `CodexAppServerClient` and rejects the
legacy `BridgeRunner` owner. The external adapter remains intentionally
separate; it is not evidence of a second primary runtime. See
[ADR 0001](../adr/0001-ha-app-runtime-ownership.md).

## 2. Transport and trust boundary

The browser uses Home Assistant administrator-authorized WebSocket and HTTP
surfaces. The Integration retains the private App bearer credential, validates
Supervisor discovery and authenticated readiness, and requires API v1 for an
App. Manual external configuration is advanced-only, explicitly negotiates v0,
and is capability-limited. See
[ADR 0002](../adr/0002-ha-origin-transport-and-trust.md).

## 3. Private state and authentication

The App initializes private Bridge and Codex-home directories under `/data`;
workspaces are constrained to `/config/workspaces`. Its Bridge token is a
mode-checked private file and the long-lived Bridge/Codex process receives a
sanitized environment. ChatGPT device authentication is owned by the App's
app-server flow. Existing external Bridge state, conversation history, and
credentials are deliberately not migrated. See
[ADR 0003](../adr/0003-private-state-and-device-auth.md).

## 4. Distribution and updates

`codex_bridge_app` is an experimental, cold-backup App definition. Its build
context is staged from repository source and a locked Codex release record.
The current workflows build versioned images, reject an existing version tag,
and define digest signature, SBOM, and provenance verification. Generated
Bridge wheels and source archives are no longer tracked under
`bridge_service/dist`; release assets are generated by the build path. See
[ADR 0004](../adr/0004-immutable-app-distribution.md).

These repository controls do not assert that an image was published or that an
automatic update, image rollback, or cold restore has succeeded in a target
Home Assistant installation.

## 5. External-Bridge retirement boundary

The external Bridge and Windows route remain private compatibility/rollback
carriers through 0.6.x. They are not stopped, deleted, migrated, or made public
by this implementation. The protected HAOS sandbox gate has passed; retirement
still requires remote-flow evidence, cold backup/restore and App update/rollback
evidence, plus explicit user removal of the VM fallback. Deletion belongs no
earlier than the next breaking release. See
[ADR 0005](../adr/0005-external-bridge-retirement.md).

## 6. Evidence and open acceptance gates

The repository contains focused App package, build-context, sandbox-contract,
Integration protocol/configuration-flow, and release-workflow tests. A
production amd64 App image also passed `/usr/local/bin/sandbox-self-test` on a
protected HAOS target: the child AppArmor profile, zero capabilities,
no-new-privileges/seccomp boundary, filesystem and mount isolation, process
environment/private-state isolation, network denial, and namespace-operation
denial all held. Authenticated readiness then passed with the runtime
attestation present.

This snapshot does **not** claim a cold backup/restore, a prior-image rollback,
an externally hosted Home Assistant session on an OpenAI-blocked network, or
completion of the external-Bridge retirement trigger. Those remain operational
acceptance gates.
